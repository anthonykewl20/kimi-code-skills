# Sample Project Structure After Setup

After running `/skill:setup-kimi-skills` in your project, you'll have:

my-project/
├── AGENTS.md                  ← Karpathy behavioral guidelines (you add this)
├── CONTEXT.md                 ← Domain language (generated by setup)
├── .kimi/
│   ├── skills-config.toml     ← Issue tracker, labels, docs paths
│   └── skills/                ← Project-specific skill overrides (optional)
│       └── my-custom-skill/
│           └── SKILL.md
├── docs/
│   ├── adr/
│   │   ├── .gitkeep
│   │   └── 20260517-remember-me-auth.md   ← ADR example
│   └── context/
│       └── .gitkeep
├── src/
│   └── ...
├── tests/
│   └── ...
└── package.json / pyproject.toml / etc.

## How Skills Load (Priority Order)

Kimi Code discovers skills in this order (first match wins):

1. `.kimi/skills/` (project-specific overrides)
2. `~/.kimi/skills/` (your personal skills — where we install)
3. `~/.config/agents/skills/` (generic agents directory)
4. Built-in Kimi skills

This means you can override any global skill with a project-specific version.

## Example: Project-Specific Override

Create `.kimi/skills/tdd/SKILL.md` in your project to customize TDD rules:

```markdown
---
name: tdd
description: TDD for THIS project — uses Vitest instead of Jest
---

# /tdd — Project-Specific Test-Driven Development

> This project uses Vitest + React Testing Library.

## Test Commands

```bash
# Run all tests
npm run test

# Run specific file
npm run test -- src/auth/auth.test.ts

# Watch mode
npm run test:watch
```

## Test Patterns

- Unit tests: `*.test.ts` alongside source files
- Integration tests: `*.integration.test.ts` in `tests/integration/`
- E2E tests: `*.spec.ts` in `tests/e2e/`
```

When you run `/skill:tdd` in this project, Kimi will load the project-specific version instead of the global one.
